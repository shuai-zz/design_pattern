# 面向对象程序设计

## 好处

1. 模块化与易维护-Modularity & Maintainability
2. 可复用性-Reusability
3. 可扩展性-Extensibility & Scalability
4. 灵活与低耦合-Flexibility & Loose Coupling

```text
Extensibilty - 扩展功能
Scalability - 扩展资源
```

## 特性

1. 封装
2. 继承
3. 多态

## SOLID原则

1. 单一职责原则(Single Responsibility Principle, SRP)  

    ```text
    一个类 / 模块只负责一项职责，避免职责混杂导致修改困难
    -> 应该只有一个引起变化的原因
    以游戏举例：
        一个游戏的设计应该将游戏界面和游戏逻辑分离
    ```

2. 开放-封闭原则(Open/Closed Principle, OCP)

    ```text
    软件实体（类、模块、函数）对扩展开放，对修改关闭（通过抽象、接口实现扩展）。
    -> 可以扩展，不能修改
    以一国两制为例：
        大陆社会主义制度不能改变（对修改封闭）；
        香港澳门长期在资本主义制度下发展，强行修改不合理（对扩展开放）
    ```

    在文章中，作者使用了一个“考研失败”的例子说明OCP原则，埋头考研放弃校招是*拒绝对扩展开放*，这个例子深深地引起了我的共鸣，考研失败确实对我影响很大，读到这段时，春招时手足无措的感觉卷土重来，希望自己引以为戒，灵活变通。由于作者说他认为这个例子不恰当，这里就不使用该例子。最后引用书中的一句话，**”希望读者您能对痛苦关闭，对快乐开放。​“**

3. 里式替换原则(Liskov Substitution Priciple, LSP)

   ```text
   子类必须能替换父类，且替换后不影响程序正确性（子类需遵循父类契约）
   -> 在软件里，把父类都替换成它的子类，程序的行为没有变化
   已Animal类为例：
        猫、狗、羊都是依赖Animal类，当实例化不同对象时，只需要更改实例化的地方，程序其他处不需要改变
   ```

4. 依赖倒置原则(Dependency Inversion Priciple, DIP)

    ```text
    高层模块不应该依赖低层模块。两个都应该依赖抽象；
    抽象不应该依赖细节，细节应该依赖抽象
    -> 要针对接口编程，不要对实现编程
    以电脑结构为例：
        cpu与主板之间通过LGA等协议相连（针对接口设计）✅
        而不是Intel CPU只能连接华硕主板（针对实现连接）❌
    ```
